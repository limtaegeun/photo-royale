## 코드에서 도출한 이벤트-크리티컬 리스크 (중점 검증 근거)

1. **구독 에러 핸들러 부재**: `subscribeToRoom`/`subscribeToParticipants`가 `onSnapshot`에 에러 콜백을 넘기지 않는다 → 세션 만료·권한 오류 시 화면 갱신이 **조용히 중단**될 수 있다 (O-01, M-03).
2. **Firestore 지연 보상(latency compensation)**: 오프라인 쓰기가 로컬 스냅샷에 먼저 반영된다 → 오프라인에서 레디가 "된 것처럼" 보이지만 서버·타 기기엔 미반영일 수 있다 (G-05, M-01).
3. **playing 전이 레이스**: rules상 participants create는 방 status와 무관하게 허용 → 시작 직후 늦게 입장한 게스트의 동선 확인 필요 (E-12, N-04).
4. **혼동 문자**: 코드 알파벳은 0/O·1/I/L 제외지만 입력 정규화는 대문자화뿐 → 사용자가 `0`을 치면 not-found로만 수렴 (A-11).
5. **비보안 컨텍스트 의존**: `crypto.getRandomValues`(방 생성)·`navigator.clipboard`(링크 복사)는 HTTPS 전제 (M-07).

## 테스트 환경·사전 조건

- 로컬 dev: `npm run dev` (Firebase env `.env.local` 필요 — Auth/Firestore 연결). 이벤트 리허설(P군)은 실제 배포 환경에서 재수행
- **계정**: ① 호스트 A, ② 게스트 B(성별 male), ③ 게스트 C(성별 female), ④ 신규 가입용 미사용 이메일, ⑤ gender 없는 계정(E-09용) — 스케일 테스트(N군)는 게스트 계정 추가 확보
- **기기 매트릭스** (M군·P군 필수): iOS Safari 실기기 1대 + Android Chrome 실기기 1대 + 데스크톱 Chrome(호스트/관제용). 인앱 브라우저(카카오톡)로 공유 링크 수신 1회 이상
- 멀티 클라이언트: 브라우저 프로필/시크릿 창 병행 + 실기기 2대 이상
- Firestore 데이터 직접 조작(레거시 데이터·status 전이 시뮬레이션)은 Firebase 콘솔(admin 권한, **rules 미적용**) 사용
- 보안 규칙 TC(J군)는 클라이언트 SDK 콘솔 스크립트 또는 rules emulator로 검증
- 네트워크 시뮬레이션: DevTools 오프라인/Slow 3G 스로틀 + 실기기 비행기 모드 토글
- 표기: **[unit]** = 단위 테스트가 이미 커버 → live-qa에서는 통합 관점 재확인만, 무표기 = live-qa 중점 대상

우선순위: **P0** 이벤트 당일 리허설(전 항목 통과 필수) / **P1** 핵심 플로우·강인성(막히면 이벤트 불가) / **P2** 주요 엣지·에러 경로 / **P3** 표시·접근성·회귀
